# Livebase v0.1 — Minimum Viable Core

## Why this file exists

`Livebase` should not become a vague promise about “better knowledge management.”

This file defines the smallest core that would make `v0.1` feel like a real work instead of:
- another note store
- another local AI shell
- another static personal wiki

---

## Core test

`v0.1` is real only if it can already do this:

# captured knowledge stays more usable over time than it would in static notes alone

That is the bar.

---

## Minimum viable core

### 1. Local-first knowledge store
The system must work locally for one person without cloud-first dependence.

This does not mean cloud sync can never exist.
It means the core identity must already stand locally.

---

### 2. Source-preserving ingestion
The system must preserve original source material.

At minimum, a captured item should retain:
- source content or source link
- capture time or relevant time context
- enough origin context to distinguish source from later abstraction

Without this, the system will drift too quickly.

---

### 3. Progressive structure on top of raw material
The system must support a lightweight structured layer without forcing heavy schema design up front.

At minimum, the user should be able to move from:
- raw note / raw source
- to linked object / recurring concept / project context

without having to design a full ontology first.

---

### 4. Reliable retrieval beyond note scavenging
The system must retrieve something more useful than file-name search or random note fragments.

At minimum, retrieval should be able to produce:
- source-grounded results
- related context when needed
- a usable working bundle for a real task

This is where context packs start to matter.

---

### 5. Durable write-back
Finished work must be able to strengthen the system.

At minimum, `v0.1` needs a path for durable residue such as:
- next-time rule
- linked note update
- project-state update
- source-linked lesson
- context-pack seed

Without write-back, the system only stores; it does not learn.

---

### 6. Anti-rot direction
The system must have at least a basic stance against decay.

At minimum, `v0.1` should make it possible to:
- detect duplicates or redundant summaries
- preserve stable links to recurring objects
- update rather than endlessly append noise
- keep source and derived layers distinguishable

Without this, the system will degrade as it grows.

---

## What is not required for v0.1

The following can wait:
- enterprise permissions
- multi-tenant architecture
- giant ontology systems
- graph visualization as a product center
- workflow automation platform behavior
- chat-first AI layer as the main identity

These may become useful later.
They are not required to prove the core work.

---

## The shortest success definition

`Livebase v0.1` succeeds if it proves:

# a local-first system can preserve source, grow light structure, retrieve dependable context, and accept durable write-back better than static notes alone

If it cannot prove that, it has not yet become a real work.
