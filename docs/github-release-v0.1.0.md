# Livebase v0.1.0 Release Notes

## Summary

Livebase v0.1.0 is the release where the project becomes explicit about what it is trying to solve.

This is not another note app, another personal wiki, or another local AI chat shell.

Livebase starts from a sharper problem:

# knowledge gets stored, but does not stay alive as durable context

v0.1.0 defines the project around a local-first response to that problem:

# keep durable context alive

That means keeping knowledge:
- callable
- connected
- source-grounded
- writable-back after real work
- usable across human-agent continuity

---

## What changed

### Product-center definition
Livebase is now framed as:

# a local durable context base focused on liveliness, not mere storage

The shift matters because the project is now judged by whether it reduces context decay, not by whether it stores notes elegantly.

---

### The real enemy is now explicit
Livebase v0.1 names the failure modes it exists to reduce:
- sink
- fragmentation
- drift
- retrieval unreliability
- missing write-back
- system rot

This makes the project more than a generic knowledge-tool concept.

---

### A local-first v0.1 boundary is now explicit
v0.1 is not trying to be:
- enterprise-first
- cloud-first
- a giant workflow platform
- a heavy ontology system
- a chat-first AI shell

It is trying to prove a narrower claim:

# a local-first system can keep durable context more alive than static notes alone

---

### Minimum viable core was defined
Livebase v0.1 now has a defined minimum viable core:
- local-first knowledge store
- source-preserving ingestion
- progressive structure
- reliable retrieval beyond note scavenging
- durable write-back
- anti-rot direction

---

### Minimum objects were defined
The repo now defines a minimal object model for v0.1:
- Source
- Note
- Project
- Entity
- Residue
- Context Pack

This keeps the structure strong enough to matter without collapsing into schema theater.

---

### Interaction and usage path were defined
Livebase v0.1 now includes:
- interaction model
- first usage path
- agent role definition
- example scenarios
- a context engineering position

This makes the repo feel much more like a usable system definition and much less like a floating thesis.

---

### Stress evidence was added
Livebase v0.1 now has three stress reads covering:
- sink
- fragmentation
- source drift
- retrieval unreliability
- missing write-back
- anti-rot pressure
- local-first erosion
- premature enterprise drift
- migration pressure
- stale context packs

Current stress result:
- batch 1: `15 / 0 / 0`
- batch 2: `15 / 0 / 0`
- batch 3: `15 / 0 / 0`

This is still contract-level evidence, not runtime system proof.
But it is enough to show the repo has a coherent center under pressure.

---

## Bottom line

Livebase v0.1.0 is the first version that begins to stand as a distinct project rather than another knowledge-base idea.

Its center is now clear:

# keep durable context alive
