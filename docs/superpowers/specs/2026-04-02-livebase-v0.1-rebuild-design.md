# Livebase v0.1 Rebuild Design

**Status:** approved for implementation

## One-line

Livebase is not AI memory. It is a local-first work continuity engine that helps humans and agents resume real tasks from the last unresolved evidence gap instead of restarting from zero.

## Product center

The center is not:
- generic memory
- chat history replay
- a broad agent platform
- a project wiki with AI attached

The center is:

# resumable work

Livebase should preserve the smallest durable state that lets the next run continue with confidence:
- current task state
- unresolved evidence gaps
- next action
- execution boundaries
- verified write-back
- durable residue

## Structural model

Livebase v0.1 should be understood as three stacked layers:

### 1. Knowledge layer

Stable source material and project context.

Objects:
- `Project`
- `Source`
- `Note`

Role:
- preserve source truth
- preserve long-lived standards
- provide grounding for future work

### 2. Work layer

The current interrupted thread of work.

Objects:
- `Task`
- `EvidenceGap`
- `Checkpoint`

Role:
- capture where the work stopped
- record what is still missing
- record what should happen next

### 3. Continuation layer

The protocol that turns saved state into the next actionable run.

Objects:
- `ContinuationPack`
- `ExecutionContract`
- `VerificationLedger`
- `Residue`

Role:
- resume work with the smallest trustworthy next step
- bound execution before the adapter starts
- require evidence-backed write-back after work completes
- strengthen the next cycle

## Product principles

1. **Continuation over recollection**
   Save what is needed to continue work, not everything that happened.

2. **Evidence gaps over summaries**
   The primary recovery primitive is the next unresolved evidence gap, not a broad recap.

3. **Contracts before execution**
   The handoff to an agent or adapter should be an execution contract, not a vague task description.

4. **Verification before write-back**
   Write-back should carry checks and observed evidence, not only outcome language.

5. **Residue should be minimal and durable**
   Only save what materially strengthens the next run.

## v0.1 core loop

`setup -> capture -> task create -> checkpoint -> resume -> contract -> act -> writeback -> strengthen`

Meaning:
- `setup`: initialize the local continuity engine
- `capture`: ingest stable source material
- `task create`: open a work thread
- `checkpoint`: stop with explicit state and evidence gaps
- `resume`: recover the next actionable continuation pack
- `contract`: turn recovery into a bounded execution contract
- `act`: human or adapter executes
- `writeback`: submit a verification ledger
- `strengthen`: save residue and updated task/project state

## Scope for this rebuild

### In scope

- local-first JSON store
- Bun + TypeScript implementation
- `src/core` + `src/cli` structure
- project/source/task/evidence/residue core objects
- `resume` returning a continuation pack
- `contract` producing a task-centric execution contract
- `writeback` accepting a verification ledger
- `setup` and `doctor` for out-of-box local use
- E2E tests for the main loop

### Out of scope

- MCP server implementation
- multi-device sync
- broad adapter ecosystem
- semantic/vector retrieval
- complex permissions/governance system
- heavy UI

## Naming shifts

To reflect the product center, v0.1 should rename two critical surfaces:

- `TaskPackage` -> `ExecutionContract`
- `WriteBackPackage` -> `VerificationLedger`

These are not cosmetic changes. They shift the system from packaging and summaries toward bounded execution and evidence-backed write-back.
