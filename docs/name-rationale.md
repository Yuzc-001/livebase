# Why the project is named Livebase

## The short answer

Because the project is not trying to be another static knowledge base.
It is trying to be:

# a base where durable working context stays alive

---

## Why not a more generic name

Names centered on:
- notes
- wiki
- brain
- memory
- vault
- graph

would pull the project back toward older categories.

The project is not centered on storage, branding, or abstract intelligence.
It is centered on a sharper promise:

# working context should remain usable over time

---

## Why “Livebase” fits

### Live
Signals that the system should resist decay.
Context should stay:
- callable
- connected
- source-grounded
- writable-back

### Base
Signals that this is not just a note surface or a chat UI.
It is the underlying local base that later work can build on.

Together, `Livebase` implies:

# a local base that keeps durable working context alive instead of letting it become static storage

---

## Bottom line

The name matters because it keeps the project pointed at the right enemy:

# not lack of storage, but loss of durable working context after storage
