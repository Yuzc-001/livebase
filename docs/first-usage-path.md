# Livebase — First Usage Path

## Why this file exists

`Livebase` should not require users to design a complete system before they can benefit.

This file defines the simplest first-usage path for `v0.1`.

---

## First-usage goal

The first user experience should prove one thing quickly:

# this system keeps a small body of knowledge more alive than static notes would

---

## Path

### Step 1 — Add one real source
Examples:
- an article
- a project note
- a meeting note
- a useful link
- a working document

Do not start with importing an entire life archive.

### Step 2 — Add one working note
Create one note that interprets, summarizes, or reacts to that source.
This establishes the difference between source and derived thinking.

### Step 3 — Attach to one project or recurring context
If the material belongs to a real project or ongoing thread, connect it.
This prevents the first objects from becoming isolated.

### Step 4 — Retrieve a context pack
Ask for a task-oriented retrieval bundle from the system.
The result should feel more dependable than random note search.

### Step 5 — Finish one piece of work and write back one residue
Examples:
- a next-time rule
- a project-state update
- a durable clarification
- a linked lesson

This proves the system is not only archival.

---

## What should happen after first use

After one good pass, the user should understand:
- source and abstraction are different layers
- projects can accumulate real context
- retrieval can produce usable working bundles
- completed work can strengthen the system

---

## What should not happen

The first-usage path should not require:
- a full ontology design
- migration of all old notes
- plugin hunting
- configuration sprawl
- complicated dashboards before value appears

---

## Bottom line

The first experience should make the system feel alive early.
If value appears only after major setup, `v0.1` has failed the opening test.
