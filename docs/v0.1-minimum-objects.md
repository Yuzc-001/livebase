# Livebase v0.1 — Minimum Objects

## Why this file exists

`Livebase` should avoid two failures:
- no structure at all
- too much schema too early

This file defines the smallest object set that is likely sufficient for `v0.1`.

---

## Core rule

Use the smallest object model that can support:
- source preservation
- retrieval
- linkage
- write-back
- anti-rot behavior

---

## Minimum object set

### 1. Source
A source is the origin-bearing object.

Examples:
- imported article
- original note
- meeting note
- captured link
- document
- transcript

Why it matters:
- preserves source truth
- anchors later abstractions
- prevents drift from derived content

Minimum fields:
- id
- title or label
- content or reference
- source type
- created/captured time
- optional project linkage

---

### 2. Note
A note is a human-usable working layer on top of or beside sources.

Examples:
- interpretation note
- working thought
- summary note
- synthesis note

Why it matters:
- supports actual thinking and use
- gives the system a flexible human-readable layer
- can remain lighter than a fully promoted object

Minimum fields:
- id
- body/content
- linked sources
- created/updated time
- optional project/entity links

---

### 3. Project
A project is a durable context container for ongoing work.

Examples:
- repo rebuild
- client delivery
- research track
- personal initiative

Why it matters:
- prevents work from fragmenting across unrelated notes
- gives retrieval and write-back a durable home

Minimum fields:
- id
- name
- status
- linked sources/notes/residue

---

### 4. Entity
An entity is a recurring thing that deserves stable identity.

Examples:
- person
- tool
- company
- concept
- document family

Why it matters:
- keeps recurring references from drifting into text-only duplication
- improves linkage and retrieval consistency

Minimum fields:
- id
- name
- type
- linked sources/notes/projects

---

### 5. Residue
Residue is the durable write-back object produced after meaningful work.

Examples:
- next-time rule
- lesson
- project-state update
- stable clarification
- linked decision note

Why it matters:
- makes the system stronger after work completes
- turns knowledge from storage into evolving memory

Minimum fields:
- id
- residue type
- content
- linked source/project/note/entity
- created time

---

### 6. Context Pack
A context pack is a task-oriented retrieval bundle.

Examples:
- current project context
- agent handoff pack
- source-grounded research bundle

Why it matters:
- converts the system from archive into callable working context
- is the bridge between stored knowledge and active work

Minimum fields:
- id
- purpose
- included objects
- generated time
- optional freshness marker

---

## Relationship sketch

```text
Source → Note
Source → Entity
Source → Project

Note → Project
Note → Entity

Project → Residue
Project → Context Pack

Entity → Residue

Context Pack → Source / Note / Project / Entity / Residue
```

This does not need to become a heavy graph product in `v0.1`.
It only needs to support dependable linkage.

---

## Promotion logic

Not every captured thing needs to become a full durable object immediately.

A good `v0.1` should allow promotion like this:
- raw source captured
- note added
- recurring thing recognized as entity
- ongoing thread promoted into project
- finished work leaves residue
- active work bundles a context pack

This keeps structure progressive instead of mandatory.

---

## Bottom line

If `Livebase v0.1` can support these six object types cleanly, it likely has enough structure to prove the core work without collapsing into either chaos or schema theater.
