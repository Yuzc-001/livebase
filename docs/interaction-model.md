# Livebase — Interaction Model

## Why this file exists

A durable working context base does not become alive merely because objects exist.
It becomes alive when humans and agents can interact with it in a stable way.

This file defines how `Livebase` should be used.

---

## Core principle

# interaction should strengthen the working context, not route around it

A weak system gets bypassed:
- users keep separate notes elsewhere
- agents answer from short-term context only and never write back
- work ends without residue entering the base

A strong system becomes the natural place where work returns.

---

## Main interaction loop

### 1. Capture
The user or agent adds:
- a source
- a note
- a link
- a document
- a project artifact
- a decision-relevant observation

This should be low-friction.
The goal is to preserve useful material without forcing heavy structure.

### 2. Connect
The system links captured material to:
- projects
- entities
- standards
- existing notes
- related sources

This can begin lightly.
Not every object needs immediate full structure.

### 3. Retrieve
The user or agent should be able to retrieve:
- source-grounded context
- connected working context
- a small dependable bundle for the task at hand

This is where context packs matter.

### 4. Work
Real work happens:
- writing
- research
- planning
- implementation
- review
- screening
- handoff

The system should support work, not interrupt it with excessive process.

### 5. Write back
After meaningful work, the user or agent writes back the smallest durable residue that improves future work.

This is how the base stays alive.

---

## Human interaction

For humans, the system should feel like:
- easy to put things into
- easy to reconnect later
- trustworthy enough to reuse
- not demanding a large setup ceremony

Humans should not need to become ontology designers just to benefit.

---

## Agent interaction

For agents, the system should support:
- source-grounded reads
- stable references to recurring objects and standards
- retrieval of context packs instead of random note fragments
- durable write-back after meaningful work

Agents should not treat the system as a decorative archive.
They should treat it as a durable working context layer.

---

## Good interaction signs

The system is behaving well when:
- important material resurfaces at the right time
- projects accumulate dependable context
- repeated work gets easier instead of noisier
- source and abstraction remain distinguishable
- both human and agent can continue work from the same durable thread

---

## Failure signs

The system is failing when:
- capture is easy but retrieval is weak
- write-back almost never happens
- users keep bypassing the system with ad hoc notes
- agents summarize but do not strengthen the base
- structure becomes heavier while usability becomes worse

---

## Bottom line

A real `Livebase` system should create a loop where capture, connection, retrieval, work, and write-back all reinforce each other.
Without that loop, the system is still just storage.
