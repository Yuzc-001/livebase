# Livebase v0.1 — Product Thesis

日期：2026-03-23  
状态：作品定义 / skill 初始中轴

---

## One-line judgment

`Livebase` should not be understood as a nicer note app or a local AI chat over files.

If this project is real, it should be understood as:

# a local durable working context base for human-agent collaboration

More sharply:

# knowledge should not merely be stored — it should remain usable as high-quality working context for future tasks and decisions

---

## Why this project should exist

The current landscape offers many adjacent capabilities:
- note-taking
- page organization
- search
- linking
- local file storage
- AI question answering over documents

But there is still a recurring gap:

# captured knowledge often decays before it can remain useful as working context

Things are written down.
Files are stored.
Links are saved.
Even summaries are generated.
But when real work resumes, the right context is often still missing.

That is the opening for this project.

---

## The real enemy

The enemy is not “lack of a note app.”
The enemy is:

# working-context decay

That decay appears in recurring forms:

### 1. Sink
Important material is captured, then disappears into storage.

### 2. Fragmentation
Projects, people, tasks, notes, documents, and standards stop remaining connected.

### 3. Drift
Abstractions and AI-generated summaries drift away from source truth.

### 4. Retrieval unreliability
Search retrieves fragments, but not dependable working context.

### 5. Missing write-back
Work ends without strengthening the next cycle of work.

### 6. System rot
The system gets heavier, noisier, and less trustworthy over time.

If the project cannot push these failures down, it is not yet a real work.

---

## The center

The center is not storage.
The center is not chat.
The center is not generic memory.

The center is:

# durable working context

That means `Livebase` should do five things well:
1. preserve source truth
2. preserve decision context and project continuity
3. let structure grow progressively
4. support reliable retrieval for action
5. let finished work write back durable residue

---

## Structural requirement

A real `Livebase` should not collapse into either:
- document storage only
- or abstract context objects with weak source grounding

Its stronger structure is:

# document layer + context layer

### Document layer
The human-readable source and working layer.
This is where source documents, markdown knowledge files, standards, project notes, handoff notes, and residue notes can naturally live.

### Context layer
The system layer that turns saved material into durable working context.
This is where identity, linking, retrieval structure, context packs, decision context, and write-back pathways should live.

The document layer keeps source truth readable and reviewable.
The context layer makes that material callable and reusable for real work.

Both are necessary.

---

## What makes this different from adjacent tools

### Not just note-taking
A note app can store content well while still letting it sink and fragment.

### Not just local RAG
A local RAG system can answer from files without preserving stable, reusable working context.

### Not just a graph
A graph-first system can become too heavy to start with.

### Not just AI knowledge chat
A chat shell can feel smart while remaining unreliable, source-drifting, and non-durable.

`Livebase` must care about:
- source grounding
- durable structure
- reliable retrieval for action
- preservation of standards and judgments
- high-value write-back
- long-horizon usability

---

## v0.1 standard

A true `v0.1` should already show:
- a sharp problem definition
- a coherent local-first core
- clear boundaries
- a path for lightweight structure to grow from real use
- a path for humans and agents to retrieve and write back reliable working context
- explicit room for a document layer, a context layer, and future execution adapters

It does not need to be a full platform.
But it must already feel like a non-empty center.

---

## What v0.1 is not

`Livebase v0.1` is not:
- an enterprise suite
- a giant permissions platform
- a workflow automation hub
- a cloud-first SaaS by default
- a mandatory ontology-building exercise

It is:

# the first serious attempt to build a local system that keeps durable working context alive instead of merely storing material

---

## Bottom line

If this project succeeds, users should feel:
- I can start locally without designing a system first
- important material stays closer to source and decision context
- retrieval is more dependable than note scavenging
- completed work strengthens the next cycle instead of bypassing it
- the system has a natural place for both readable documents and operational context

That is the standard that makes `Livebase` worth existing.
